# Contract Management UX Upgrade Plan

## Summary
The goal is to revamp the Contract Management UX to be highly mobile-friendly, auto-populate signature details from profiles (enforcing DB validation), consolidate the status tracker into a single scannable line, format the financial breakdown precisely, and allow inline editing directly on the full-page contract document.

## Current State Analysis
- Contract text is a single raw string with literal `_______` placeholders for signatures.
- Edits are done via a bulky accordion form at the bottom of the screen.
- Status is shown via two grid cards which take up vertical space.
- The UI is not fully optimized for touch devices (small buttons, cramped scroll areas).

## Proposed Changes

### 1. Backend: Signature Auto-Population & Validation
- **File**: `server/services/contract.service.ts`
  - **Action**: Update `generateContractFromSnapshot` to validate that both the Artist and Organizer profiles contain required fields (`legalName` or `name`, `panNumber`, `permanentAddress`). Throw an explicit error (`Missing required profile information for contract generation`) if the DB lacks these details.
- **File**: `server/contract-utils.ts`
  - **Action**: Update `generateContractText` to pre-fill the `Name:` and `Title:` fields dynamically from the DB snapshot. Replace the blank lines for `Digital Signature`, `Date`, and `IP Address` with unique placeholder tokens (e.g., `[[ARTIST_SIGNATURE]]`, `[[ARTIST_DATE]]`, `[[ARTIST_IP]]`).
- **File**: `server/routes/contracts.ts`
  - **Action**: In the `/contracts/:id/sign` endpoint, after recording the signature, perform a string replacement on `contract.contractText` to swap the placeholder tokens with the actual signature data, timestamp, and IP address, then save the updated text to the DB.

### 2. Frontend: Full-Page Document & Inline Editing
- **File**: `client/src/pages/contract/ContractPage.tsx`
  - **Action**: 
    - Remove the `max-h-[300px]` constraint on the document viewer to let it flow naturally full-page.
    - Write a parser to split `contract.contractText` into sections using regex `/(?=\n\d+\.\s+[A-Z])/`.
    - Render each section as a distinct UI block. If a section corresponds to an editable category (e.g., "1. EVENT DETAILS", "2. TRAVEL", "5. HOSPITALITY"), render an inline "Edit" icon button floated to the top-right of that text block.
    - Clicking the inline edit button opens a mobile-friendly Drawer/Sheet (or Dialog) specifically for that category's fields, removing the need for the bulky accordion form at the bottom.

### 3. Frontend: Single Line Status Tracker
- **File**: `client/src/pages/contract/ContractPage.tsx`
  - **Action**:
    - Remove the `PartyStatusCard` grid.
    - Create a new horizontal scrolling flex container (`overflow-x-auto whitespace-nowrap`).
    - Implement the requested format:
      `Artist (You) • Reviewed ✓ • Edit Used ✓ • Accepted ✓ • Signed ✓ │ Promoter • Reviewed ✓ • Edit Used ⏳ • Accepted ⏳ • Signed ⏳`
    - Use appropriate icons and colors (Green for done, Gray/Muted for pending) to make it highly scannable on mobile.

### 4. Frontend: Financial Breakdown UI Update
- **File**: `client/src/pages/contract/ContractPage.tsx`
  - **Action**:
    - Update the financial box to match the exact requested layout:
      ```
      Financial Breakdown
      Gross Booking Value        ₹X
      Platform Fee (Y%)         -₹Z
      Net Payout                 ₹W
      ```
    - Ensure the math correctly reflects the `commissionBreakdownJson` generated by the category rules.

### 5. Mobile & UX Best Practices
- **File**: `client/src/pages/contract/ContractPage.tsx`
  - **Action**:
    - Use full-width cards with large touch targets.
    - Make the Contract Document text responsive (adjust font size for mobile, ensure no horizontal overflow).
    - Stick the main action buttons (Sign, Accept, Walk Away) to a fixed bottom bar (`sticky bottom-0 bg-background/95 backdrop-blur z-50 p-4 border-t`) so they are always accessible on mobile without scrolling.

## Assumptions & Decisions
- **Assumption**: The database schema does not need to change to support the signature placeholders; it's entirely handled via text manipulation in the backend routes.
- **Decision**: To achieve "inline edits" on a raw string, we will split the string visually in React based on the numbered clauses. This avoids changing the underlying data structure while vastly improving the UX.

## Verification
- Ensure contract generation blocks if user profiles are incomplete.
- Verify that placeholders are correctly swapped out with actual signature data upon signing.
- Test the UI on mobile viewport to ensure horizontal scrolling works for the tracker and bottom actions remain sticky.
